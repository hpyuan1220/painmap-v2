## §0.5 v2.1 變更 — 蘇格拉底大一統

v2 解決了階段混淆，但階段一內部仍殘留三個不一致：

1. **0–25 Pain Quality Score**：5 維度 × 1–5 分為了「訓練判斷力」而存在，但分數本身會被誤用為綠燈（「24/25，去做 App」），於是設計者加了 disclaimer 解釋「分數不是答案」——**修補一個本不該存在的欄位**。
2. **教學模式 vs 產出模式**：雙模式為了「在 R&D 場域隱藏分數」而存在；保護的是一個本不該存在的欄位，再用 400 行的 mode 切換守住界線。
3. **TRIZ 六矛盾**：借用發明創新理論的學術權威，包裝一個其實是通用 trade-off 識別的機制——名實不符；使用者強迫把現實塞進 6 個分類。

第一性原理重新看：**寫作本身就是反思**。9 步 worksheet 要訓練的是「觀察 → 拆解 → 證據 → 判斷」的書面紀律；不需要分數，不需要模式，不需要分類學。蘇格拉底式工具不打分數，只問問題。

v2.1 對應措施：

| 移除 | 替代 |
| :--- | :--- |
| 5 維度 1–5 分（人群具體度／頻率／強度／不滿／證據） | 維持為**思考視角**寫進 prompt copy，但**不評分、不存欄位** |
| 0–25 Pain Quality Score | 100 字書面理由 + judgment + 最有把握/最沒把握 + 下一步 |
| 教學模式 vs 產出模式 | 單一輸出：使用者親筆書面身份證 |
| TRIZ 六矛盾分類學（卡 5） | 使用者用自己的話寫 side_a / side_b / 哪邊被犧牲 + 一句話原因 |
| 「過關 / 退回」措辭 | 中性反思提示「再想想看」/「回去把 X 想清楚再來」 |

v2.1 後本文件保留的核心方法論：9 階段流程、anti-IdeaCheck 立場、觀察→拆解→證據→判斷的訓練弧、SECI 三段使用、第一性原理物理量 #1 / #2、S1 / S2 詰問。

## §1 設計原則

### 1.1 為什麼要強制拆兩階段（v2 的最大變革）

v1 的 7 步流程把「痛點發想」與「商業驗證」混為一個 sprint，導致：

- **小白受眾被嚇跑**：第一次發想就要架收款連結、設計手作交付
- **AI 引導被破壞**：使用者一進入流程就在想 App / 收費 / 商業模式，正是 Sunny 講稿明令禁止的反模式
- **判斷力沒被訓練**：「真假痛點」這個獨立判斷被「能不能賺錢」蓋過

v2 強制分階段，原因如下：

| 階段 | 主問題 | 主方法 | 主產出 |
| :--- | :--- | :--- | :--- |
| **階段一** | 這是真痛點還是假痛點？ | AI 證據蒐集 + 真人訪談規劃 + 自我判斷 | 書面「真/假」判斷 |
| **階段二** | 能不能用這個痛點建立生意？ | 手作交付 + 預售 + 72 小時 sprint | 第一筆真實付款 |

兩個階段**不可逆向**：
- 階段一沒過 → 階段二沒意義（驗證假題目）
- 階段一過 → 階段二**未必過**（真痛點未必是好生意）

### 1.2 為什麼把框架重組為「鏡頭+引擎」（v2.1 修訂）

並列多個框架時，操作者必須自己決定「先用哪個看」，這是定義漂移的溫床。v2 維持鏡頭管線設計，v2.1 進一步把 TRIZ 從鏡頭名單移除（理由見 §0.5）：

| 理論 | 階段歸屬 | 角色 |
| :--- | :--- | :--- |
| **SECI** | 階段一 | 鏡頭——從口語抱怨到證據組合到真人確認 |
| **trade-off 自陳** | 階段一 | 鏡頭——使用者用自己的話寫出「兩件事不能同時要」（**不套用任何分類學標籤**） |
| **BCG DRI** | **階段二** | 鏡頭——驗證後決定攻擊層次（**不在發想階段使用**） |
| **第一性原理** | 跨階段 | 引擎——還原成物理量；#1、#2 在階段一，#3、#4 在階段二 |
| **蘇格拉底** | 跨階段 | 引擎——S1 定義 / S2 壓力測試貫穿全程 |

**為什麼 TRIZ 從鏡頭名單移除：** TRIZ 在工程設計領域有效，但在「找痛點主人翁的取捨」這個任務上，它的 6 個矛盾標籤反而把使用者的真實 trade-off 強塞進預設分類（「個人化 vs 規模化」聽起來很學術，但訪談對象用不出這種詞）。v2.1 改成讓使用者用自己的話寫——同樣訓練「找出兩件事不能同時要」的能力，但不引入借用權威的分類學。

### 1.3 BCG DRI 為什麼不在階段一

DRI 回答「**怎麼解**」——這必須在「痛點是真的」確認後才有意義。在階段一就跳到 DRI，會把使用者拖進「想 App / 想商業模式」的反模式。

**唯一例外：** 階段一 Step 9 的真假判斷後，可以**用 1 句話標注**「如果是真的，初判可能屬於 D/R/I 哪一層」作為階段二的入口提示——但**不參與真假判斷**。

### 1.4 為什麼禁止跳步、禁止套用分類學

**跳步反模式：** 聽到抱怨直接想「這要 Reshape」「這要做 App」，跳過 SECI 外化與 trade-off 自陳 → 把 AI 該怎麼用當成痛點。

**分類學反模式（v2.1 修訂）：** 把使用者寫出來的 trade-off 強塞進 TRIZ 六矛盾或任何 N×M 分類框，因為「有標籤看起來比較專業」→ 分類學壓垮現實，訪談對象認不出自己。v2.1 在卡 5 直接移除 TRIZ 六矛盾選項，從源頭斷掉這條反模式。

引擎擋這兩個反模式：S1 強迫先定義；S2 強迫檢查定義漂移；卡 5 只接受使用者的 prose 描述。

## §2 雙引擎（不重新定義，只組裝）

### 2.1 Elon 第一性原理 = 還原器

**完整定義見：** `first_principles_playbook.md §2`

本流程的物理量**分階段使用**：

| 物理量 | 一句話 | 階段 | 檢核時機 |
| :--- | :--- | :--- | :--- |
| **#1 具體的人** | 你能當下打電話給 3 位 | 階段一 | Step 2（5 要素檢核） |
| **#2 正在投入時間/成本的痛** | 替代方案有名字 | 階段一 | Step 2（5 要素檢核） |
| **#3 手工可交付的動作** | 今晚就能做一次給 1 個人 | **階段二** | sprint manual Hour 12 |
| **#4 付款通道** | 現在就有可分享的收款連結 | **階段二** | sprint manual Hour 24 |

**v2 重要修正：** v1 把 #2 寫成「正在**付錢**的痛」，會誤導發想階段就找付費客戶。v2 改為「正在**投入時間/成本**的痛」——只要 workaround 有名字、有時間/情緒/金錢成本投入，就成立。

### 2.2 蘇格拉底 = 詰問器

**完整定義見：** `socratic_first_principles_rnd_workflow.md §2, §3`

本流程使用 S1/S2 兩個工序，**每個鏡頭前後各跑一次**：

- **S1（鏡頭開始前）**：誰／何時／如何觀察到——三要素必齊。
- **S2（鏡頭結束前）**：
  1. **可證偽性**：什麼證據會讓我放棄這個痛點？
  2. **假因排除**：是不是其實是另一個痛點？
  3. **定義漂移**：起初定義 vs 現在定義是否同一件事？

## §3 鏡頭詳解

### 3.1 鏡頭 1：SECI（橫跨整個階段一）

SECI 不再是一次性鏡頭，而是**三段使用**：

| SECI 工序 | 階段一位置 | 動作 |
| :--- | :--- | :--- |
| Socialization | Step 1 + Step 8 | 入口聽抱怨；訪談規劃時的真人現場確認 |
| Externalization | Step 2 + Step 3 | 抱怨變痛點 5 要素；卡關公式句 |
| Combination | Step 6 + Step 7 | AI 蒐集公開證據；自己先猜+對照 AI 整理 |
| Internalization | **階段二** | 透過 sprint 把驗證結果變成下一輪的判斷力 |

**誤用警告：**
- ❌ 用 AI 生成合成 persona 取代 Socialization
- ❌ 在沒有具體場景時就跳到 Externalization

### 3.2 鏡頭 2：trade-off 自陳（階段一 Step 4，v2.1 取代 TRIZ）

把痛點主人翁的處境翻譯成「兩件事不能同時成立」的書面語言——**不用任何預設分類標籤**，只用使用者自己的話。

**結構（不是分類學）：**

```
他想要 ___（side_a，使用者親筆 ≥10 字）
但他也想要 ___（side_b，使用者親筆 ≥10 字）
他通常會放掉 [a / b]
為什麼放掉那邊？___（一句話原因）
```

**為什麼移除六矛盾（v2.1）：** 借用 TRIZ 工程矛盾標籤包裝通用 trade-off 識別，反而讓使用者把現實塞進 6 個預設格子。改成 prose 自陳後，訪談對象能認出「這就是我說的話」，而不是「這是哪一條 TRIZ contradiction」。

**誤用警告：**
- ❌ 把 trade-off 標籤當痛點本身（trade-off 是診斷視角，不是答案）
- ❌ 為硬套框架強塞 → 寫不出 side_a/side_b/sacrificed_reason 就退回 Step 2 補 5 要素，**不是**換一個 trade-off 名稱

### 3.3 鏡頭 3：BCG DRI（**階段二** — 不在階段一使用）

**理論：** Deploy（補丁）/ Reshape（流程）/ Invent（品類）三層 AI 應用。

**為什麼在階段二：** 這是「**怎麼解**」的判斷，必須在痛點被確認真實之後才有意義。

**階段一 Step 9 的角色：** 只在「真痛點」判定後，**附 1 句**初判 D/R/I 層次作為階段二入口，**不參與真假判斷**。

**階段二完整應用：** 見 `painmap_disruption_framework.md`。

## §4 階段一：痛點發想 9 步流程（不談錢、不打分數）

> **使用方式（v2.1 修訂）：** 每一步都有**輸入 / 動作 / S1 詰問 / 物理量檢核 / 輸出 / 反思檢核**。反思檢核不是「過/不過」的閘門，而是「這一題是否寫到位」的書面提醒——沒寫到位時，UI 顯示中性反思提示（「再想想看」），而不是紅色 ✗。

### Step 1. 聆聽（SECI / Socialization 入口）

| 欄位 | 內容 |
| :--- | :--- |
| **輸入** | 一句口語抱怨（聽到的、不是解釋的） |
| **動作** | 把原句逐字寫下，不改寫；附上時間、地點、來源 |
| **S1 詰問** | （無） |
| **物理量檢核** | （無） |
| **輸出** | 原句逐字稿 + 1 個具體場景證據 |
| **反思檢核** | 原句中可指出**至少一個具體可命名的人**——若只剩匿名代號，UI 會問：「這幾個字像不像是你自己幫他想的？」 |

### Step 2. 抱怨變痛點 5 要素（SECI / Externalization + S1 + 物理量 #1、#2）

| 欄位 | 內容 |
| :--- | :--- |
| **輸入** | Step 1 逐字稿 |
| **動作** | 填 5 要素表 + 列 3 人名單 + 寫 workaround 名稱 |
| **S1 詰問** | 痛點主人是誰／何時／如何觀察到？三要素齊嗎？ |
| **物理量檢核** | **#1**：3 人名單（有名字、聯絡方式、背景）<br>**#2**：他現在用什麼 workaround？這個 workaround 有具體名字嗎？ |
| **5 要素表** | 具體人群／具體場景／發生頻率／明確成本／現有解法不滿 |
| **輸出** | 5 要素表 + 3 人名單 + workaround 名稱 |
| **反思檢核** | 5 要素都具體（無「應該」「大概」）；3 人名單成立；workaround 有名字。蘇格拉底提示：「你能不能今天就傳訊息給其中一個？如果不能，那他算「有名字的人」嗎？」 |

### Step 3. 卡關公式（SECI / Externalization 命題化）

| 欄位 | 內容 |
| :--- | :--- |
| **輸入** | Step 2 5 要素 |
| **動作** | 改寫成「我每次要 ___，都會卡在 ___」單一句子（使用者親筆，可選用 AI clarifying questions 迴圈幫忙澄清） |
| **S1 詰問** | 「你寫的公式句，能不能被一個外人聽完就重複給第三個人？」 |
| **物理量檢核** | （已在 Step 2 完成 #1、#2） |
| **輸出** | 卡關公式句（≥30 字、具體、無抽象名詞） |
| **反思檢核** | 句子裡的兩個空格是否具體；如果其中一個還是「想要更好的工作流」這種抽象詞 → 蘇格拉底提示「再回去問主人翁一次」 |

### Step 4. 自陳 trade-off（v2.1 取代原 TRIZ 步驟）

| 欄位 | 內容 |
| :--- | :--- |
| **輸入** | Step 2 5 要素 + Step 3 卡關公式 |
| **動作** | 用使用者自己的話寫：「他想要 side_a」+「他也想要 side_b」+「他通常放掉哪邊」+「為什麼放掉那邊（一句話）」 |
| **S1 詰問** | 「他想要 ___，但又同時想要 ___。如果他能放掉其中一邊，他不會卡在這裡——所以他**放不下哪邊**？」 |
| **禁止** | 套用 TRIZ 標籤、寫「個人化 vs 規模化」這種抽象名詞、複製預設選項 |
| **物理量檢核** | （無） |
| **輸出** | side_a / side_b / sacrificed (a or b) / sacrificed_reason |
| **反思檢核** | side_a / side_b 各 ≥10 字使用者自己的話；sacrificed_reason 是一個具體句子；若寫不出 → 退回 Step 2 補 5 要素，**不是**換一個分類標籤 |

### Step 5. 選 AI 工具角色

| 欄位 | 內容 |
| :--- | :--- |
| **輸入** | Step 3 卡關公式 + Step 4 trade-off |
| **動作** | 從工具地圖選 1 個 AI（ChatGPT Deep Research / Claude / Perplexity / Gemini） |
| **判準** | 第一次發想 → ChatGPT Deep Research；已有大量文字 → Claude；要查最新趨勢 → Perplexity / Gemini |
| **輸出** | 選定的 AI 工具 |
| **反思檢核** | 知道你為什麼選這個工具（1 句話） |

### Step 6. 用 AI 找證據（SECI / Combination 加速）

| 欄位 | 內容 |
| :--- | :--- |
| **輸入** | Step 5 選定工具 + Step 1–4 全部產出 |
| **動作** | 複製 §7 的 8 題證據蒐集 prompt，**明確禁止 AI 設計產品/商業模式、明確禁止 AI 打分數** |
| **S1 詰問** | （無——這一步是執行） |
| **物理量檢核** | （無） |
| **輸出** | AI 整理的 8 題回覆 |
| **反思檢核** | AI 沒進入解決方案；AI 沒主動產出 1–5 分；8 題都有答 |

### Step 7. 自己先猜，再讀 AI 回覆（蘇格拉底自我論述前置）

| 欄位 | 內容 |
| :--- | :--- |
| **輸入** | Step 6 AI 回覆（**先不看**） |
| **動作** | 先寫 3 個猜測（誰最痛／哪場景最痛／可能假痛點）；**phase A 鎖**確保猜測在讀 AI 之前完成；再讀 AI 回覆並對照差異 |
| **S1 詰問** | 「AI 給的 8 個答案哪個讓你最意外？」「我猜 vs AI 答的差異在哪？AI 補了什麼？AI 漏了什麼？」 |
| **物理量檢核** | （無） |
| **第二輪 prompt** | 請 AI 整理成「痛點判斷表」，挑出最值得優先研究的 1 個人群（**不請 AI 打分**） |
| **輸出** | 猜測表 + 對照表 + 痛點判斷表 |
| **反思檢核** | 4 個檢查點都檢視過；能寫出 3 個差異點 |

**為什麼要先猜：** 直接看 AI 回覆會被它牽著走（「啊它說的對！」是順著問題寫的錯覺）。先猜後對照才能訓練判斷力。Card 7 是其他卡片的學習對象——把 phase A → AI → phase B 的鎖視為蘇格拉底模板。

### Step 8. 規劃真人訪談（SECI / Socialization 現場確認）

| 欄位 | 內容 |
| :--- | :--- |
| **輸入** | Step 7 痛點判斷表 |
| **動作** | 從 AI 給的 5 種訪談對象挑 2 種；寫 3 個訪談題；列訪談禁則 |
| **S1 詰問** | 「我為什麼選這 2 種而非其他 3 種？」「這 3 道題，你今晚就能傳給其中一個人嗎？」 |
| **物理量檢核** | 回頭看 #1：訪談對象是不是 Step 2 3 人名單裡的人？ |
| **訪談禁則** | 不推銷解法、不問「你會付多少」、只問現況 |
| **輸出** | 訪談對象清單 + 3 訪談題 + 訪談禁則確認 |
| **反思檢核** | 至少 1 位有具體聯絡管道；3 訪談題不是推銷題 |

### Step 9. 真/假痛點判斷（階段一終點，v2.1 極簡化）

| 欄位 | 內容 |
| :--- | :--- |
| **輸入** | Step 1–8 全部產出 |
| **動作** | 書面回答「真痛點 / 假痛點 / 待訪談」+ 100 字理由 + 最有把握的證據 + 最沒把握的地方 + 下一步 |
| **S1 詰問** | （已在 Step 7 完成自我論述） |
| **S2 壓力測試** | 1. 什麼證據會讓我放棄？<br>2. 是不是其實是另一個痛點？<br>3. Step 1 vs Step 8 是同一人同一情境嗎？ |
| **5 個蘇格拉底自問**（純螢幕提示，不存欄位） | • 你能說出 3 個有名字的人嗎？<br>• 你看到他每週遇到幾次？是猜的還是有人告訴你？<br>• 他付出最多的是時間、錢、心力還是關係？<br>• 他現在解法最讓他不爽的點，能用他的話寫出來嗎？<br>• 最有把握的證據 vs 最薄弱的環節分別是什麼？ |
| **可選**（不參與真假判斷） | 1 句話初判 D/R/I 哪一層作為階段二入口 |
| **輸出** | 書面身份證（judgment + reason_100w + most_confident_evidence + least_confident + next_action） |
| **反思檢核** | 三個 S2 都有書面答案；100 字理由達標；下一步明確；**沒有任何欄位是 1–5 分數**。若 reason 不足 100 字 → UI 顯示「再想想看，這 100 字是要說服未來的你」 |

**v2.1 極簡化重點：**
- **刪除** 5 維度 1–5 分（人群具體度／頻率／強度／不滿／證據）
- **刪除** 0–25 總分
- **刪除** 分數帶建議（20–25 排訪談 / 15–19 縮人群 / 14 以下換題）
- 5 維度只當作螢幕上的**自問句提示**（pure UI），不寫進資料層

**階段一終點的 3 種出口：**

```
□ 真痛點   → 進入階段二（Step 8 訪談 → sprint manual）
□ 假痛點   → 換題目重來（這就是 worksheet 的價值——省下 3 個月走錯路）
□ 待訪談   → 先完成 Step 8 訪談再回來更新 Step 9
```

## §5 單一輸出（v2.1 取代雙模式）

v2.1 之前本節描述「教學模式（25 分鏡子）vs 產出模式（4 階標籤）」雙模式切換。v2.1 把分數整個刪除後，雙模式失去存在理由——保留之物只剩 lifecycle 標籤。

### 5.1 為什麼移除雙模式

雙模式存在的唯一原因是「在 R&D 場域隱藏 0–25 分數」。一旦分數移除（理由見 §0.5），「教學版要顯示分數、產出版要藏分數」這個切換就沒有變數可切換。保留模式切換 = 保留沒有用途的複雜度。

### 5.2 單一輸出格式：書面身份證

階段一終點產出**一份**書面交付：

```
痛點：[Step 2 痛點句]
判斷：[真痛點 | 假痛點 | 待訪談]

書面理由（≥100 字，使用者親筆）：
[把 Step 1–8 的證據連回判斷的邏輯]

最有把握的證據：[使用者親筆]
最沒把握的地方：[使用者親筆]

S2 壓力測試：
1. 可證偽性：什麼證據會讓我放棄？
2. 假因排除：是不是其實是另一個痛點？
3. 定義漂移：Step 1 主人翁 vs Step 8 訪談對象同一人/情境嗎？

下一步：[排訪談 / 補證據 / 換題目 / 進階段二 sprint]
完成日期：[YYYY-MM-DD]
```

**沒有 0–25 分。沒有 5 維度評分。沒有 teaching/production toggle。** 判斷的權威來自使用者親筆把證據連回判斷的書面邏輯，不來自一個數字。

### 5.3 Lifecycle 標籤（不是評分）

引用 `painmap_prd.md §3 Epic 2`，4 階標籤是**生命週期狀態**而非評分：

| 狀態 | 進入條件 | 階段 |
| :--- | :--- | :--- |
| `draft` | Step 1 完成 | 階段一 |
| `structured` | Step 1–9 全部通過、判定真痛點 | 階段一 |
| `verified_interview` | 階段二 sprint Hour 24–48：≥ 5 位訪談 + 紅隊報告 | 階段二 |
| `verified_payment` | 階段二 sprint Hour 48–72：≥ 1 位真實付款 | 階段二 |

**鐵律（引自 `painmap_moat_design.md M3`）：** 任何階段**禁止輸出 0–25 分數或任何 N×M 評分鏡子**。違反 = 違反 painmap 反焦慮品牌定位。

## §8 反模式（Anti-Patterns）

### 8.1 跳階段反模式（v2 新增）

**症狀：** 痛點還沒判定真假，就跳到「設計手作交付」「架收款連結」「跑 72 小時 sprint」。

**根因：** 把階段一與階段二混為一個流程。

**解方：** 階段一 Step 9 退出條件鎖死「書面真假判斷」；沒判定真痛點不能進階段二。

### 8.2 跳步反模式

**症狀：** 跳過 Step 7（先猜後對照），直接讀 AI 回覆並接受。

**根因：** 把 AI 當答案而非證據。

**解方：** Step 7 phase A 鎖強制要求書面猜測；phase A 未完成不解鎖 AI 對照欄位。

### 8.3 分類學反模式（v2.1 取代「單鏡頭過度」）

**症狀：** 把每個痛點的 trade-off 硬塞進 TRIZ 6 矛盾或其他 N×M 預設標籤。

**根因：** 借用框架的學術權威 = 看起來比較專業。

**解方（v2.1）：** Step 4 直接移除 TRIZ 6 標籤選項；只接受使用者親筆 prose。Step 9 S2 的「假因排除」仍要寫至少 1 個替代假設並排除——但用使用者自己的話，不用框架名稱。

### 8.4 評分反模式（v2.1 修訂自原「IdeaCheck 病」）

**症狀：** 任何形式的數字評分出現——不論是 1–5、0–25、A/B/C 級別、還是「Pain Quality Score」。

**根因：** 數字看起來客觀，但會被誤用為綠燈（「24/25，可以做了」）；任何 disclaimer 都擋不住這個誤用。

**解方（v2.1）：** §0.5 把分數整個刪除。書面理由 100 字 + 親筆 most_confident / least_confident 取代分數作為判斷力的訓練載體。如果有人問「分數怎麼算」，回答：「沒有分數。寫 100 字理由說服未來的你。」

### 8.5 取悅 AI 反模式

**症狀：** 問 AI「這痛點 OK 嗎」，AI 給正向回饋後就行動。

**根因：** AI 是類比機器，會基於相似成功案例給正向回饋。

**解方：** Step 9 三個 S2 詰問**禁止用 AI 直接生成答案**——你必須自己寫。100 字理由也必須親筆。

### 8.6 合成 persona 反模式

**症狀：** 用 AI 生成虛構受訪者跳過真人訪談。

**根因：** Step 1 沒有真人來源。

**解方：** Step 1 反思檢核鎖「至少一個具體可命名的人」；Step 8 訪談對象必須在 Step 2 3 人名單內。

### 8.7 Pass/Fail UI 反模式（v2.1 新增）

**症狀：** UI 把每個 Step 的反思檢核做成「✗ 未過關」紅色提示、「過關條件 3/5」進度條。

**根因：** 把蘇格拉底反思做成考試評等。

**解方：** UI 用中性 `ReflectionHint`（pending / thinking / ok）取代 pass/fail 圖示；文案用「再想想看」「回去把 X 想清楚再來」取代「未過關」「退回卡 X」。詳見 `docs/painmap_worksheet/design/components/exit_gate_check.md`。

## §9 附錄 — 補習班老師案例完整走完階段一 9 步

### Step 1. 聆聽

**逐字原句**（2026-04-15 與林老師訪談）：
> 「我每週六晚上要寫 30 個學生的家長回報，平常週間都要記筆記但常漏，到週末翻 7 次小考成績單、翻群組對話、翻學生作業，常寫到半夜兩點。寫完還要怕語氣太冷或太重，傷家長感情。」

### Step 2. 5 要素 + 物理量 #1、#2

**痛點句：** 「我每週六要寫 30 則家長回報，都會卡在資料散在週間 7 次小考、要寫得具體、不能傷感情。」

**5 要素：**
- 具體人群：補習班國中部老師（30–60 學生規模）
- 具體場景：每週六晚上 21:00–02:30
- 發生頻率：每週 1 次（每年 50+ 次）
- 明確成本：5–6 小時/週 + 半夜情緒成本
- 現有解法不滿：LINE+Excel 拼湊（找資料慢）；Notion 模板（貼來貼去更慢）；ChatGPT 寫稿（罐頭感）

**物理量 #1（3 人名單）：** 林老師 / 王老師 / 陳老師（皆有名+聯絡方式）
**物理量 #2（workaround 名稱）：** LINE + Excel 成績表 + 翻群組對話

### Step 3. 卡關公式

「我每次要**在週末把週間散在 LINE / 小考 / 群組對話的資訊收成 30 則家長回報**，都會卡在**要寫得具體又不能傷感情、但只有 2–3 小時可寫**。」

### Step 4. 自陳 trade-off（v2.1：使用者親筆，不套標籤）

- **side_a（他想要的）：** 家長能在 LINE 上看到「我的孩子」被個別關照——具體事件、具體進步、具體擔心；不要罐頭訊息。
- **side_b（他也想要的）：** 一週只用 2–3 小時把 30 個學生的家長訊息寫完，剩下時間給家人和備課。
- **通常放掉：** side_a（罐頭訊息）
- **為什麼放掉那邊：** 因為 side_b 是體力上限——半夜 2 點睡覺已經是極限，再個別化就活不下去。

### Step 5. 選 AI 工具

**選定：** ChatGPT Deep Research（第一次發想，需要外部證據）

### Step 6. 用 AI 找證據

執行 §7 prompt，AI 回覆 8 題（略，含 5 個工具評論、3 個論壇貼文、2 篇補教產業文章證據）。

### Step 7. 自己先猜+讀 AI

**我的猜測：**
- 最痛的人：補習班導師（負責全班）
- 最常場景：段考後一週
- 可能假痛點：也許大型補習班用班務系統已解決

**AI 補了什麼：** 「行政老師」也很痛（之前沒想到——他們要彙總所有導師的回報）
**AI 漏了什麼：** 沒提到「家長很急 vs 老師寫不完」的緊張感（我從訪談聽到的）
**痛點判斷表：** 優先研究「中小型補習班導師」，理由是大型已被系統解決、行政老師是次級訴求。

### Step 8. 真人訪談規劃

**優先訪談對象：**
1. 王老師（台中北屯，已聯絡）
2. 陳老師（高雄三民，已聯絡）

**3 訪談題：**
1. 你最近一次寫家長回報是什麼時候？花多少時間？
2. 你現在用什麼方法整理週間資料？
3. 有沒有試過任何工具或服務？為什麼放棄？

**訪談禁則確認：** 不推銷解法、不問會付多少。

### Step 9. 真/假判斷（v2.1：書面，不評分）

**判斷：** 真痛點（待訪談 2 人後再升級為 `verified_interview`）

**書面理由（100 字）：** 三個物理量物件都成立——具體人 3 位、workaround 有名字（LINE+Excel）、痛苦成本明確（5–6 小時 + 情緒）。林老師訪談裡說了「我寧願寫到 2 點也不要罐頭」，這把 trade-off 從假設變成事實。最大不確定是大型補習班是否已被系統解決，下一步用王、陳老師訪談排除。

**最有把握的證據：** 林老師親口說「半夜 2 點睡」是穩定行為（不只一次）、3 個 workaround 全部被她試過再棄用（LINE+Excel / Notion / ChatGPT），這個棄用清單比任何問卷都有說服力。

**最沒把握的地方：** 沒驗證大型補習班是否已被內部系統解決。如果王、陳老師其中一個說「我們有班務系統」，那痛點主人翁可能要縮小到「沒系統的中小型補習班」。

**S2 壓力測試：**
- 可證偽性：王、陳兩位都說「我寧願自己寫，習慣了」→ 放棄此痛點。
- 假因排除：「也許老師真痛是怕說壞消息」→ 排除（林老師明說是時間黑洞，不是措辭）。
- 定義漂移：Step 1 林老師 vs Step 8 王、陳老師——同類型同情境 ✓。

**下一步：** 排王、陳訪談（本週內），訪談完更新 lifecycle 狀態到 `verified_interview`，再考慮進階段二 sprint。

**初判 D/R/I（不參與真假判斷）：** Reshape——光加 AI 寫稿助手是補丁，需重設「老師如何在週間記錄」的工序。

## 結語

> AI 時代的 0→1，最稀缺的不是工具、不是模型——是**判斷力**。
> 判斷力分兩種：判斷「真假痛點」與判斷「能不能賺錢」。
> 兩者**不可混淆**——混在一起就是 v1 的錯。
> v2 的修正是強制拆階段：階段一只訓練判斷真假，階段二才驗證商業。
> 階段一的篩網不通過 → 不進階段二。
> 階段一通過 → 階段二未必過，但你已經省下 90% 的瞎做時間。
