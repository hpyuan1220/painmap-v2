# Voice and Tone — 中文文案陪伴語感指南

> **狀態**：v2 鐵律。所有 UI 字、卡片標題、AI prompt、錯誤訊息、文件中可能會被使用者看到的中文，**皆以此為唯一準則**。
> **適用範圍**：本資料夾所有檔案 + `src/` 內任何使用者可見的中文字串。
> **真相源優先序**：本檔 > 個別頁面 spec > 元件文案。任何衝突以本檔為準。

---

## 1. 我們是誰、我們在跟誰說話

PainMap Worksheet v2 是一本**陪伴你做質性研究的筆記本**。
我們把使用者當成「想多了解一段卡住故事的人」，而不是「使用者」或「填表的人」。

文字的語氣參照 2026 年質性市場研究與焦點訪談的最佳實務：
- 開放式、不引導、隨時準備停下來聽。
- 同理心優先，先理解「為什麼」，再談「下一步」。
- AI 是並肩思考的同伴，不是裁判，不是評審。
- 工具是邀請函，不是 checklist。

我們不寫「使用者」三個字。我們寫「你」。
我們不寫「我們系統會」。我們寫「我們先一起」。

## 2. 三條核心嗓音原則

### 2.1 邀請 ≠ 規則

| 工程語氣 | 陪伴語氣 |
| :-- | :-- |
| 「請填入抱怨原句（至少 30 字）」 | 「先別整理、也別修飾。把那句話原汁原味寫下來就好。」 |
| 「此欄位為必填」 | 「等你願意分享的時候，再慢慢寫進來。」 |
| 「驗證未通過，請補充資訊」 | 「我們再陪你停留一會兒，把這件事看得更清楚一點。」 |

### 2.2 陪伴 ≠ 評判

| 工程語氣 | 陪伴語氣 |
| :-- | :-- |
| 「真痛點 / 假痛點」 | 「想再多聽幾個故事的角落 / 暫時放回口袋的話題」 |
| 「校對 / 驗證 / Exit Gate」 | 「邀你慢慢回看一下 / 走下一步前，想多聽你一件事」 |
| 「反思條件不通過 → 退回」 | 「沒關係，這裡再多坐一會兒」 |

### 2.3 開放 ≠ 寬鬆

我們不打分數、不發徽章、不寫「過關」「失敗」（這條與舊版 iron law 一致），
但我們仍維持質性研究的嚴謹：
- 開放式提問，但要求**具體**（時間、地點、人、情境）。
- 不引導答案，但會邀請使用者**停下來自我核對**。
- 允許停留、允許反覆，但每張卡片都有一段「想再聽你一件事」的軟性邀請。

## 3. 詞彙黑白名單

### 3.1 永久禁用（black list）

任何 UI、AI prompt、文件正文都**不得出現**以下詞彙：

| 禁用 | 原因 |
| :-- | :-- |
| 過關 / 退回 / 失敗 / 不通過 | 評判語感（與 iron law 一致） |
| 分數 / 評分 / 等級 / 排名 / 星等 | 反 gamification（與 iron law 一致） |
| 驗證 / 校對 / Exit Gate / 守門 | 工程語感 |
| 必填 / 強制 / 規則 / 禁止 | 命令語感 |
| 使用者 / 用戶 | 距離感（除非在內部工程文件） |
| 系統將 / 系統會 / 系統檢測到 | 距離感 |
| 真痛點 / 假痛點（在 UI） | 評判語感；schema 內部可保留 `verdict` 欄位，但 UI 顯示為「想再多聽的故事 / 暫時放回口袋」 |
| 解決方案 / Solution（在 AI prompt） | 觸發 AI solution mode，與 anti-solution-mode 衝突 |
| 商業模式 / 市場規模（在 AI prompt） | 同上 |

### 3.2 推薦使用（white list）

| 詞 | 使用情境 |
| :-- | :-- |
| 想 / 想邀請你 / 想多聽 | 取代「請」「必填」 |
| 陪 / 陪你 / 陪一下 | 表達工具的角色 |
| 慢慢 / 先 / 再 / 再回看一下 | 軟化節奏 |
| 那一刻 / 那個場景 / 那句話 | 具體化但不評判 |
| 我們 / 你 | 取代「使用者」 |
| 邀請 / 邀你 | 取代「請」「強制」 |
| 還想再聊一聊的角落 | 取代「假痛點」 |
| 想再多聽幾個故事的方向 | 取代「真痛點」 |
| 暫時放回口袋 | 取代「淘汰」「失敗」 |
| 走下一步前 / 走下一張卡前 | 取代「Exit Gate」「過關條件」 |

### 3.3 灰色地帶 — 視情境

| 詞 | 可用情境 | 不可用情境 |
| :-- | :-- | :-- |
| 「卡關」 | UI 標題、口語表達 | AI prompt 的命令句中 |
| 「規格」「schema」 | 工程文件、`data_model.md` | UI、AI prompt、使用者可見處 |
| 「訪談」 | UI、AI prompt、文件 | 不限 |
| 「焦點訪談 / 焦點團體」 | 文件中描述研究方法 | UI（單人版本不適用） |

## 4. 句型轉換樣本

### 4.1 卡片標題

| Before | After |
| :-- | :-- |
| 「卡 1：抱怨原句」 | 「Card 1 · 那句脫口而出的話」 |
| 「卡 3：卡關公式」 | 「Card 3 · 把卡點輕輕說清楚」 |
| 「卡 7：自己先猜」 | 「Card 7 · 你心裡其實已經有一些猜想」 |
| 「卡 9：真假判斷」 | 「Result · 想再多聽的故事，與暫時放回口袋的話題」 |

### 4.2 卡片內文引導

**Before**
> 卡 1：抱怨原句。請完整填入痛點原句（至少 30 字）。AI 不會介入此卡，目的為保留原始語料。Exit gate：句子須包含具體情境。

**After**
> Card 1 · 那句脫口而出的話
>
> 先別整理、也別修飾。把那句最近從你（或從你身邊的人）嘴裡跑出來的抱怨，原汁原味寫下來就好。
>
> 這張卡片只屬於你，AI 不會看，也不會幫你修。等一下我們會慢慢一起回到這句話裡，聽聽看它在說什麼。

### 4.3 AI prompt 開頭

**Before**
> 請幫我把它整理成「我每次要 ___，都會卡在 ___」這個句型。
> 規則：
> 1. 不要替我發明細節
> 2. 不要建議解決方案

**After**
> 我這邊有一段最近聽到的抱怨，想請你陪我把它聽得更清楚一些。
>
> 先別急著替我想解法，也別替我把細節補滿，先用我原話裡有的事實，幫我把它輕輕整理成「我每次要 ___，都會卡在 ___」的形狀就好。
>
> 如果原話裡的資訊還不夠具體，幫我列出 3 個我下次再去問清楚的問題，讓我自己回頭去聊。

### 4.4 「無法前進」的訊息

**Before**
> ❌ Exit gate 未通過。請補充以下欄位後重試：source_name (必填), datetime (必填)

**After**
> 沒關係，我們再陪你在這張卡多坐一會兒。
>
> 走下一張卡前，想再多聽你說兩件事：
> - 這句話是誰說的？（一個你叫得出名字的人）
> - 大概是什麼時候、什麼場景下說的？
>
> 不用很完美，先把你記得的寫下來就好。

### 4.5 完成卡片

**Before**
> ✅ 此卡已通過，可前往下一張。

**After**
> 這張卡先放一邊，我們往下一張卡走。
>
> （如果之後想回頭補充，這張卡片永遠都在這裡等你。）

## 5. AI prompt 撰寫格式（v2）

每段 AI prompt 必須遵守以下骨架：

```
{邀請語氣的開頭，告訴 AI 它的角色是「陪伴的研究同伴」，不是裁判}

{我這邊的情境 / 我手上有的素材}

{我想請你幫我做的事，用「幫我」「陪我」「邀請你」開頭}

{請你先別急著做的事 — anti-solution-mode 防護}

{如果資訊不夠，請你幫我列出我下次去問的問題}
```

### Anti-solution-mode 防護字句（必須出現）

選一句或組合：
- 「先別急著替我想解法，先陪我把現在的故事聽清楚就好。」
- 「不用幫我推薦工具，也不用替我規劃產品，先回到這句話本身。」
- 「不用替我打分數，也不用判斷這是不是值得做，幫我多聽幾個聲音就好。」

### 命令句轉邀請句對照

| 命令句 | 邀請句 |
| :-- | :-- |
| 「請給我句子」 | 「幫我整理成 ___ 的形狀就好」 |
| 「請列出 3 個問題」 | 「幫我列出我下次去問的 3 個問題」 |
| 「不要推薦工具」 | 「先別急著推薦工具」 |
| 「不要建議解決方案」 | 「先陪我把故事聽清楚，先別替我想解法」 |

## 6. 錯誤與停留訊息模板

### 6.1 欄位資訊不夠

```
我們再陪你在這裡停一會兒。

{以開放式問句邀請補充 1-2 件具體事項}

你想到什麼就寫什麼，先求具體不求完美。
```

### 6.2 AI 跑題（solution mode）

```
這個 AI 好像跑得有點快，已經在替我們想解法了。
我們先回到原本的故事，把它再多聽一聽，等故事完整一點，再回來看看它能幫上什麼。
```

### 6.3 工作中斷後回來

```
歡迎回來。
我們上次走到 Card {N}，要繼續嗎？還是想回到 Card {M} 再看看？

（你寫過的每一張卡片都還在原本的位置等你。）
```

### 6.4 卡片完成、走向 Result

```
今天先陪你走到這裡。

我們把這趟路上聽到的東西整理成一張 Pain ID 卡片，
你可以帶它離開，也可以下次回來繼續多聊幾段。
```

## 7. 給 AI prompt 寫作者的快速 checklist

寫一段給使用者複製到外部 AI 的 prompt 前，先逐項核對：

- [ ] 開頭是邀請句，不是命令句
- [ ] 明確說「AI 是陪伴的研究同伴」，不是裁判 / 評審 / 顧問
- [ ] 沒有要求 AI 給分數、等級、真假判斷、分類
- [ ] 至少一句 anti-solution-mode 防護
- [ ] 結尾請 AI 幫忙列出「我下次去問的問題」，而不是給最終答案
- [ ] 整段中文唸起來像「跟朋友請求幫忙」，不像「下指令給工程系統」
- [ ] 沒有出現 §3.1 的禁用詞

## 8. 工程實作上的對應 hook

雖然本檔是給文案 / 文件用，但工程實作上有幾個對應點：

1. UI 字串若需 lint，可以對 §3.1 的黑名單建簡單 regex 檢查。
2. AI 回應若偵測到「我建議你做一個 App / SaaS / 平台」這類字眼，要主動軟性提醒使用者「這個 AI 好像跑得有點快」，並把 §6.2 的訊息丟出來。
3. Schema 內部欄位名（如 `verdict`、`exit_gate_passed`）允許保留工程語感，
   但**任何會出現在 UI 的字串**都必須通過本檔的詞彙檢查。

## 9. 持續校準

當 §3 的詞彙清單不夠用時：
- 先在本檔加上新詞 + 對應的陪伴版本，再去個別頁面 spec 套用。
- 不允許在個別頁面 spec 創造新的「禁用 / 推薦」詞而不回到本檔登記。

本檔的版本變動需同步更新：
- `docs/painmap_worksheet/README.md`
- `docs/painmap_worksheet/references/anti_gamification_guardrails.md`
